# NixOS config architecture

Flake-based NixOS config for a Framework 13 laptop. Originally forked from Ryan
Yin's config (see `flake.nix` description) and heavily customized.

## Top level

```
flake.nix          # inputs + nixosConfigurations (Aragorn, Gandalf, simple)
flake.lock
hosts/<host>/      # per-machine: configuration.nix, home.nix, hardware-configuration.nix
modules/nixos/     # system modules
modules/home_manager/  # home-manager (per-user) modules
artifacts/         # non-Nix config files & assets referenced by modules (dotfiles, scripts, keys, sops)
images/            # wallpapers etc.
docs/              # this file
```

## Flake

`flake.nix` defines a `mkNixosSystem { hostName, modules, homeModules }` helper that
calls `nixpkgs.lib.nixosSystem`. Key points:

- Two nixpkgs inputs: pinned `nixpkgs` (nixos-25.05) and `nixpkgs-unstable`.
  `pkgs-unstable` is passed through `specialArgs`/`extraSpecialArgs`, so modules can
  pull individual packages from unstable (see `maxlang.nix` — codex, claude-code,
  obs-studio, etc. come from `pkgs-unstable`).
- `specialArgs` injects `pkgs-unstable`, `inputs`, `user` ("maxlang"), `hostName`.
- home-manager is wired as a NixOS module (`useGlobalPkgs`, `useUserPackages`); each
  host supplies `homeModules`.
- Three hosts: **Aragorn** (the laptop, primary), **Gandalf**, **simple**.

## Module composition

`hosts/Aragorn/configuration.nix` imports a handful of `modules/nixos/*` plus
`maxlang.nix`, which is the **main aggregator** — it imports the bulk of system
modules (user, desktop apps, navidrome, syncthing, sops) and defines the big
user-package list. `bwlang.nix` is an alternate user profile (currently not imported).

Lots of imports are toggled by commenting/uncommenting single lines (the config is
actively "tested in production"), e.g. in `maxlang.nix` and `Aragorn/configuration.nix`.

Convention: anything that is a plain config file for some app (hyprland, rofi, yazi,
waybar, scripts…) lives under `artifacts/` and is referenced from the relevant module,
rather than being generated by Nix.

## Secrets — sops-nix

- **Module:** `modules/nixos/sops.nix` is self-contained (it imports
  `inputs.sops-nix.nixosModules.sops` itself), so secrets are enabled/disabled by the
  single `./sops.nix` import line in `maxlang.nix`. The flake input lives in `flake.nix`
  (two lines, commentable).
- **Store:** `artifacts/sops/`
  - `.sops.yaml` — recipients/creation rules (PUBLIC age keys; committed).
  - `secrets/secrets.yaml` — encrypted secrets (committed).
  - `age/keys.txt` — the age PRIVATE key (**gitignored**, never commit).
  - `PLAINTEXT-REFERENCE.md` — convenience plaintext copy (gitignored).
- **Key file path** in `sops.nix` must be absolute (`/etc/nixos/artifacts/sops/age/keys.txt`);
  sops-nix activates from `/`, so a relative path silently fails.
- **Consuming a secret:** `config.sops.secrets."<name>".path` gives a runtime file
  (under `/run/secrets/`). For services that need a *value* inline (e.g. navidrome's
  config is rendered to the store), use a `sops.templates."x".content` with
  `config.sops.placeholder."<name>"` and feed the rendered file in as a systemd
  `EnvironmentFile` — see `navidrome.nix`.
- Secrets currently managed: `cmu-pass`, `navidrome-lastfm-apikey`,
  `navidrome-lastfm-secret`, `wireguard-private-key`, `wireguard-preshared-key`.

### Editing secrets

```
cd /etc/nixos/artifacts/sops
nix-shell -p age sops --run 'SOPS_AGE_KEY_FILE=age/keys.txt sops secrets/secrets.yaml'
```

Add a recipient: add its public key under `keys:` in `.sops.yaml`, then
`sops updatekeys secrets/secrets.yaml`.

## Apply changes

```
sudo nixos-rebuild switch --flake /etc/nixos#Aragorn
# validate only:
nixos-rebuild dry-build --flake /etc/nixos#Aragorn
```
